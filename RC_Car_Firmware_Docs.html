Master Firmware Flow

This is the top-level control diagram for the whole Arduino sketch. It begins at power-on, runs setup(), then repeats loop() forever.

Complete runtime entry flow
  • This is the real project entry flow today.
  • The board always starts in obstacle mode unless the code is changed.
  • The follow-me function exists in the switch but does not contain working logic yet.

Obstacle Avoidance State Machine

This is the most important runtime diagram in the project. It shows the actual state path inside runObstacleAvoidance().

Active obstacle logic
  • The state values seen in logs are real: 0 = MOVE, 1 = SEARCH, 2 = TURN.
  • ESCAPE and STOP exist in the enum, but they are not active states in the current logic path.
  • The emergency backoff branch runs before the regular state switch when the object is too close.

Bluetooth Command Flow

This diagram shows exactly how one Bluetooth character is interpreted by readCommand().

Input parser
  • The current live sketch reacts to single characters only.
  • Manual movement happens after sending M, then sending motion letters like F, B, L, or R.
  • Follow-me can be selected by mode, but its behavior is still unfinished.

Reset Diagnosis Flow

This flowchart turns the debug lesson from the logs into a practical troubleshooting path.

Hardware diagnosis
  • logs.txt showed repeated startup banners, so a reset definitely happened.
  • logs2.text showed only one startup banner when the heavy power load was removed.
  • That result strongly points to power distribution and electrical load, not ordinary flow-control logic.
